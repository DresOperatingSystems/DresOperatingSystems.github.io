May 15, 2026 - AOSmium WebView v2.1.0 Released, MicroG Module Next

The AOSmium WebView Magisk module has been rewritten from the ground up and released as v2.1.0.

The previous v1.0.0 flashed cleanly on Motorola Moto G32 and similar devices but bootlooped Pixel 9 and several LineageOS builds because it relied on pm install at flash time (the data partition is not yet mounted at that stage on some LineageOS builds) and hid the OEM WebView via Magisk .replace markers, leaving devices without a fallback provider when AOSmium failed to load. Later versions removed the bootloop but never actually activated AOSmium, because the bundled overlay APK was plain text XML rather than a properly compiled binary AXML overlay, and the overlay targeted com.android.webview instead of the framework android package where config_webview_packages actually lives.

v2.1.0 is a complete redesign of the activation pipeline. The overlay is now a properly compiled RRO signed with v1, v2, and v3 schemes, with the AXP.OS ECDSA certificate embedded as a base64 DER block in the signature element of config_webview_packages.xml. The AOSmium APK is placed in the systemless tree at system/product/app/AOSmiumWebView/ via Magisk magic mount, satisfying the framework MATCH_FACTORY_ONLY scan without any pm install. Activation runs in service.sh after sys.boot_completed via cmd webviewupdate set-webview-implementation, with a settings put global webview_provider fallback write and a dumpsys verification step.

Two layers of bootloop safety were added. post-fs-data.sh drops a boot_pending marker on every boot which service.sh clears on success; a stale marker on the next boot means the previous boot crashed before activation completed, and the module auto-disables itself by touching disable. service.sh also flips the module into inert mode on any activation failure to prevent retry storms.

The module has been confirmed working on the following devices:

  • Motorola Moto G32 on LineageOS Android 15
  • Motorola ThinkPhone on Stock Android 15
  • Motorola Moto G7 Plus on Stock Android 10
  • Motorola Moto G7 Plus on LineageOS Android 15
  • Samsung Galaxy A05s on Stock Android 10

x86 and x86_64 are not supported because AXP.OS does not publish WebView builds for those ABIs.

With AOSmium WebView now stable, work begins next on the MicroG module (dresosmicrog). This will replace Google Play Services with MicroG for app compatibility without Google's tracking infrastructure, all in a single Magisk flash. Updates to follow in this tab as development progresses.

Released Modules

dresoswv v2.1.0
DresOS AOSmium WebView
Replaces Android's system WebView with AOSmium - a Chromium fork hardened with GrapheneOS and Vanadium security patches. The system WebView is the browser engine used internally by hundreds of apps whenever they render web content. Google's version phones home on every render. This module replaces it in a single Magisk flash.

How it works: a static RRO is placed in the systemless overlay partition to add AOSmium plus the AXP.OS signing certificate to config_webview_packages, the AOSmium APK is dropped into the systemless system tree via Magisk magic mount, and after boot complete service.sh runs cmd webviewupdate set-webview-implementation to promote AOSmium to the active provider. Activation is verified via dumpsys. Two layers of bootloop safety sit on top: a post-fs-data sentinel that auto-disables the module if a previous boot crashed, and an inert mode flag that prevents retry storms on activation failure.

Confirmed working devices:
  • Motorola Moto G32 on LineageOS Android 15
  • Motorola ThinkPhone on Stock Android 15
  • Motorola Moto G7 Plus on Stock Android 10
  • Motorola Moto G7 Plus on LineageOS Android 15
  • Samsung Galaxy A05s on Stock Android 10
Requirements: Magisk 24.0 or newer, Android 10 through 15, arm or arm64.

After flashing: reboot, then verify with adb shell dumpsys webviewupdate. The active provider should be org.axpos.aosmium_wv.

We originally wanted to release our own system but due to all these new cyber laws and the pressure Google is putting on firmware swaps we just could never release it. So we hope our project can bring our organisation back to its core values and release an idea we had once in a different light.

The DresOS Android project has been fully merged into a single comprehensive guide. The old two-repo method (Android-Degoogling and Android-opsec) is now combined into one unified system with a full app suite, security architecture document, and changelog.

Build Your DresOS Android System

Follow the unified guide below. It covers everything from unlocking the bootloader and flashing Magisk, to the full OPSEC stack with InviZible Pro, AFWall+, and the complete DresOS defensive app suite.

What the System Includes

  • Part 1 - DeGoogling - Full root method: TWRP, Magisk, Noogle microG, AOSmium WebView via Magisk module, Shizuku, system debloat, FOSS app replacements

How This System Protects You

  • Removes all Google services and tracking from the device
  • Replaces system WebView with AOSmium via Magisk module - GrapheneOS hardened Chromium
  • Routes all traffic through Tor via InviZible Pro proxy and root mode
  • Encrypts all DNS queries via DNSCrypt enforced at kernel level via iptables
  • Blocks in-app trackers via DuckDuckGo App Tracking Protection
  • Kernel-level firewall via AFWall+ with per-app rules and mobile data proxy redirect
  • AES-256 file encryption via Amaze File Manager with biometric lock
  • Scans every link before opening via URL Check
  • Scans every APK before installing via Hypatia (ClamAV signatures)
  • End-to-end encrypted email via Tuta Mail with Duck Address aliases so your real address is never exposed
  • GPS spoofing via Fake Traveler
  • MAC address randomisation per Wi-Fi network
